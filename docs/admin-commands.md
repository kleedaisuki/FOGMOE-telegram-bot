# 部署管理员使用说明

> 受众：机器人部署配置 `identity.administrator.user_id` 指定的唯一 Telegram 用户。
> 本文是运维参考，不会被普通的 `/help` 命令展示。

## 权限与使用边界

下列操作均以发送者的 Telegram user ID 与部署配置中的管理员 ID 做严格相等比较；它们不是 Telegram 群管理员权限，也不因命令所在群而获得权限。未授权请求不会执行读取、公告、发币或监控控制等副作用。

平台运维命令目前可在机器人可接收命令的任一聊天中使用；银行、账单和订阅命令另有“仅私聊”边界。若群聊内存在命令归属歧义，可显式指定机器人：`/命令@kleek_RoPL_bot`。

## 平台运维

| 命令 | 参数 | 功能 | 结果与边界 |
|---|---|---|---|
| `/stats [1..50]` | 可选整数，默认 `20` | 查看全局运行统计 | 返回用户总数、启用关键词/验证/垃圾控制/图表功能的群数，以及每类最多指定数量的群 ID 样本；最近用户固定至多 10 个。 |
| `/logs [1..200]` | 可选整数，默认 `50` | 查看当前日志文件末尾 | 只读取文件尾部，单次读取最多 64 KiB，输出会进一步限制在 Telegram 安全长度内；日志不存在时会明确提示。日志可能含敏感运行信息，勿转发到公开群。 |
| `/admin_announce <正文>` | 去除首尾空白后为 1–3500 个字符 | 创建全局公告 | 原子冻结受众并异步、可恢复地投递；命令回复只表示已入队，不等同于已送达。 |
| `/start_test_monitor` | 无 | 为**当前聊天**启动 BTCUSDT 事件合约价格模式监控 | 已在运行时不重复启动；应在希望接收监控消息的聊天中执行。 |
| `/stop_test_monitor` | 无 | 停止 BTCUSDT 事件合约价格模式监控 | 监控未运行时只返回状态，不产生额外副作用。 |

`/stats` 与 `/logs` 的参数必须是单个十进制正整数；例如 `/stats 0`、`/logs many`、`/stats 20 extra` 都会被拒绝。

## 银行、支付与权益

银行和 Billing（账单与权益）是两条故意不相连的业务链：银行只管理游戏内 `Free`（免费金币）与隔离保留的 `Paid legacy`（历史付费金币）；Billing 只管理受控支付渠道的原生金额、订单、权益（entitlement）和订阅（subscription）。**支付金额绝不兑换为金币，金币也不能兑换为支付金额。**

下列命令均仅限私聊；管理员命令除私聊限制外，还会再次严格校验部署配置中的管理员 ID。

| 入口 | 适用者 | 参数 | 结果与边界 |
|---|---|---|---|
| `/admin` | 部署管理员、私聊 | 无 | 并行汇总注册用户数与最多 5 条待审 `Free` 申请，并给出审核、发行、奖池、公告和诊断的下一步命令；仅读取，不会改动账户或资产。 |
| `/bank` | 已注册用户 | 无 | 查看 `Free` 和 `Paid legacy` 的分离余额。`Paid legacy` 是历史迁移余额，不是新的可售商品。 |
| `/request_tokens <数量> <用途>` | 已注册用户 | 正整数；用途 1–500 字符 | 创建一笔待审核的**免费金币**申请，并以 durable outbox（持久化发件箱）异步提醒部署管理员私聊。 |
| `/recharge <数量> <用途>` | 已注册用户 | 与 `/request_tokens` 相同 | 仅是免费金币申请的简写；同样会产生管理员审核提醒；不是充值、付款确认、卡密兑换或人工入账请求。 |
| `/bank_review <申请ID> approve\|reject [说明]` | 部署管理员 | UUID；可选说明最多 500 字符 | 管理员提醒会给出这条显式命令；批准时以一条可审计、平衡的账本分录发行 `Free`，拒绝时不发生资金移动。成功审核后申请人会收到异步终态回执。申请者不能审核自己的申请。 |
| `/bank_issue <用户ID> <数量> <审计用途>` | 部署管理员 | 正用户 ID、正整数、用途 1–500 字符 | 直接发行 `Free`，仅应用于经授权的运营/修复场景；必须填写可追溯用途。不能以它模拟付款、订阅或向 `Paid legacy` 入账。 |
| `/bank_fund_activity <数量> <审计用途>` | 部署管理员 | 正整数、用途 1–500 字符 | 从发行账户向可验证随机活动奖池显式注资；活动结算本身绝不自行铸币。返回注资后的奖池余额与不可变账本分录 ID。 |
| `/billing`、`/billing_order <报价ID> [续费订阅ID]` | 已注册用户 | UUID | 查看有效权益或创建 `awaiting_payment` 订单；创建订单不代表已付款。 |
| `/refund <订单ID> <原因>`、`/subscription_cancel <订阅ID>` | 订单/订阅所有者 | UUID；退款原因最多 1000 字符 | 发起退款审核，或请求在当前订阅周期结束时取消。 |
| `/billing_fulfill <订单ID>` | 部署管理员 | UUID | 仅在支付事件已由受控渠道验证并记录后履约，授予对应权益/订阅；不会写入银行账本或金币钱包。 |
| `/billing_refund_review <退款ID> approve\|reject [说明]` | 部署管理员 | UUID；可选说明最多 1000 字符 | 记录退款审核；实际渠道退款仍需由受控支付事件验证和结算。 |

没有 `$FOGMOE` 买入、兑换、swap 或卡密充值命令，也没有“管理员看见转账截图后手动把付款换成金币”的路径。`Paid legacy` 不得与 `Free` 合并、互换、作为活动押注或群组贡献；所有新的游戏内流通从 `Free` 开始，并经银行账本记录。

银行审核提醒只会投递到配置的部署管理员私聊；请让该管理员先向 Bot 发送 `/start`，否则 Telegram 可能永久拒绝主动私聊。提醒本身没有批准按钮，不会导致资金变更；管理员应核对申请 ID、申请人、金额、用途和 `/bank_pending` 的最新状态后，再显式发送一条 `/bank_review ... approve` 或 `/bank_review ... reject` 命令。

## Agent 银行工具与确认边界

在与 FogMoe 的私聊中，Agent 可以直接查询自己的余额、由管理员查询待审申请，或为当前认证用户创建一笔待审核的免费金币申请。它们不会直接改动钱包余额。

只有三类会实际改变账户或资产的 Agent 操作会生成一张私聊确认卡：审核申请、直接发行 `Free`、向活动奖池注资。确认卡只引用服务端保存的操作；点击者、私聊、管理员资格、过期时间和最终业务状态都会在执行时重新校验。批准后才执行，取消或过期不会改动资产；若进程在执行途中中断，后台会以同一业务幂等键安全恢复。

支付服务仅接受**受控的原生支付渠道**所提供且通过验证的事件。当前文档不宣称 Telegram Stars 已自动启用；如果某部署日后接入 Stars 或其他原生渠道，必须先完成该渠道的认证、事件验签/核验、幂等去重和退款事件处理，才可调用履约流程。

## 公告语义

`/admin_announce` 的正文是完整的命令剩余文本，例如：

```text
/admin_announce 今晚 22:00（UTC+8）将进行短暂维护，预计五分钟恢复。
```

接受公告时，系统在同一个数据库事务中冻结受众快照：已登记的 Telegram 用户，以及已出现在群功能配置或规范群消息投影中的群。后续新注册的用户或新加入的群不会自动加入这一次公告。

每个受众都有独立的持久化投递记录；后台 worker 使用 lease（租约）与 fencing（隔离令牌）恢复重试。所有受众进入成功或失败终态后，机器人会回复到发出命令的聊天/Topic，报告总受众数、成功数和失败数。因此：

- 不要因暂未看到最终报告而重复发送同一通知；先等待投递完成。
- 同一 Telegram Update 的重放使用稳定幂等键（idempotency key），不会创建第二份公告；回复会标注为 replay。
- 重新手动发送一条新的 Telegram 消息会产生新的 Update，并被视为新的公告请求。
- 公告文本会投递到用户私聊和符合快照条件的群聊；应避免包含密钥、内部链接、个人资料或其他不应大范围传播的信息。

## 操作建议

1. 先用 `/stats` 估计群功能覆盖面和公告影响范围。
2. 公告正文先在私聊中审校，确保链接、时区与措辞无误；发送后受众快照和正文均已持久化，不能通过再次发送来“编辑”原公告。
3. 等待终态报告；若有失败数，结合 `/logs` 诊断 Telegram 投递、权限或网络问题。
4. `/logs` 仅用于短时排障。它的内容可能含内部路径、异常上下文或用户标识，不应复制到公共频道或非受信任系统。
5. 批准 `/bank_review` 或使用 `/bank_issue` 前，核对申请 ID、用户 ID、数额和用途；把审批依据保留在受控系统中。不要把银行命令当作外部付款的兑换工具。
6. `/billing_fulfill` 前核对订单状态和已验证的渠道事件；退款审核与渠道退款结算是两个独立、可审计的状态变迁。
7. 启动 BTC 监控前确认当前聊天确实是预期的接收位置；停止操作作用于正在运行的监控，而非仅撤销某一条提示消息。

## 与群管理员命令的区别

部署管理员与 Telegram 群管理员是两套独立身份：部署管理员不会自动获得任一群的管理员资格；群管理员也不能读取全局统计/日志、发布全局公告、审核银行申请或控制 BTC 监控。

群管理员可在自己管理的群中使用 `/verify`、`/spam`、`/keyword`、`/chart bind`、`/chart clear` 与 `/resetgroup` 等群级配置功能。其中 `/resetgroup` 只清空当前群的共享记忆，不影响个人记忆、User Profile 或其他群。详见 [记忆架构说明](memory-architecture.md)。

群组小镇也遵循这条身份边界：`/town` 仅可在群聊或超级群中执行；群成员可用 `Free` 贡献，群主/群管理员可创建和建成项目。个人冒险 `/adventure` 则只允许私聊，不能把个人角色或材料放入群小镇范围。
